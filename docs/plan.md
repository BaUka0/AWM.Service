# План добавления недостающей логики по воркфлоу (Этапы 3–11)

Этапы 1 (Утверждение НР) и 2 (Утверждение периодов) — уже реализованы.

---

## Этап 3: Формирование направлений ДП/ДР

**Реализовано:** CRUD направлений, submit/approve/reject/requestRevision, доменные события.

**Не хватает:**
1. **Валидация периода** — при создании и submit направления проверять, что период "Формирование направлений" открыт (использовать `IPeriodValidationService.IsStageOpenAsync`).
2. **Уведомления** — при смене статуса направления (утверждено/отклонено/на доработку) отправлять уведомление НР через `INotificationService`. Обработчики доменных событий (`DirectionApprovedEvent`, `DirectionRejectedEvent`, `DirectionRequiresRevisionEvent`) должны отправлять уведомления.

---

## Этап 4: Формирование тем ДП/ДР

**Реализовано:** CRUD тем, approve/close, BulkApproveTopics, привязка к направлению.

**Не хватает:**
1. **Валидация периода** — при создании темы проверять, что период "Формирование тем" открыт.
2. **Статус "Черновик" для темы** — воркфлоу предполагает, что тема сначала сохраняется как черновик, затем пакетно отправляется на кафедру. Сейчас у Topic нет workflow-состояния (только `IsApproved`). Нужно добавить команду `SubmitTopicsForApproval` — пакетная отправка тем на утверждение кафедрой.
3. **Уведомления** — при утверждении/отклонении темы уведомлять НР.

---

## Этап 5: Выбор тем студентами

**Реализовано:** CreateApplication, AcceptApplication, RejectApplication, WithdrawApplication, GetAvailableTopics.

**Не хватает:**
1. **Валидация периода** — при подаче заявки проверять, что период "Выбор тем студентами" открыт.
2. **Ограничение количества заявок студента** — студент может подать только одну активную заявку (или ограниченное число). Нужна проверка в `CreateApplicationCommandHandler`.
3. **Уведомления** — НР получает уведомление о новой заявке; студент — о решении НР.
4. **Автозакрытие темы** — при достижении лимита участников (все слоты заняты) тема автоматически закрывается.

---

## Этап 6: Согласование тем

**Реализовано:** BulkApproveTopicsCommand, создание StudentWork при AcceptApplication.

**Не хватает:**
1. **Сводный запрос по согласованию** — Query, который показывает кафедре: общее кол-во тем, кол-во принятых заявок, темы без студентов, темы с превышением лимита. (`GetTopicCoordinationSummaryQuery`)
2. **Пометить тему как неактуальную** — команда для пометки тем без студентов как неактивных.
3. **Завершение согласования** — команда `CompleteTopicCoordination`, которая фиксирует итоги, блокирует изменения (закрывает все темы, финализирует привязки), уведомляет участников.

---

## Этап 7: Утверждение периодов предзащит, проверок и защиты

**Реализовано:** CreateCommission, AddCommissionMember, Expert/Reviewer entities.

**Не хватает:**
1. **Управление периодами предзащит** — команда для установки периодов "Предзащита 1", "Предзащита 2", "Защита" (возможно расширить существующий ApproveInitialPeriods или создать `SetDefensePeriodsCommand`).
2. **Автоматическое распределение студентов по комиссиям** — команда `DistributeStudentsToCommissions`, которая равномерно распределяет StudentWork'ы по созданным комиссиям предзащиты.
3. **Автосоздание расписания слотов** — команда `GeneratePreDefenseSlots`, которая создает 30-минутные слоты на указанную дату для комиссии и распределяет по ним студентов.
4. **Назначение экспертов на проверки** — команда `AssignExpertsCommand` для привязки экспертов к конкретным видам проверок и кафедре (нормоконтроль, ПО, антиплагиат).
5. **Добавление рецензентов** — команда `AssignReviewersCommand` для назначения внешних рецензентов на конкретные StudentWork'ы.
6. **Уведомления** — при утверждении периодов уведомить студентов (расписание), членов комиссий, экспертов.

---

## Этап 8: Предзащиты

**Реализовано:** SchedulePreDefense, SubmitPreDefenseGrade, RecordAttendance, FinalizePreDefense (max 3 попытки).

**Не хватает:**
1. **Загрузка материалов для предзащиты** — валидация, что студент загружает именно презентацию и черновик (проверка типа `AttachmentType` при загрузке в состоянии PreDefense).
2. **Предварительное согласование оценок** — команда `PreliminaryReconciliationCommand` (секретарь запускает этап, все члены видят оценки друг друга и могут пересмотреть).
3. **Формирование ведомости предзащиты** — команда/запрос `GeneratePreDefenseProtocol` для создания PDF ведомости предзащиты (аналогично GenerateProtocol для GAK, но для предзащиты).
4. **Запрос неуспешных студентов** — `GetFailedPreDefenseStudentsQuery` для формирования списка студентов, направленных на пересдачу.
5. **Утверждение периода пересдачи (предзащита 3)** — команда `ScheduleRetakePreDefense` для пересдач.
6. **Блокировка перехода к следующему этапу** — валидация, что студент прошел предзащиту перед переходом к проверкам (проверка в TransitionState или в SubmitForCheck).

---

## Этап 9: Проверки

**Реализовано:** SubmitForCheck, RecordCheckResult (NormControl/SoftwareCheck/AntiPlagiarism), Review upload.

**Не хватает:**
1. **Последовательность проверок** — антиплагиат доступен ТОЛЬКО после прохождения нормоконтроля. Добавить валидацию в `SubmitForCheckCommandHandler`: если `CheckType == AntiPlagiarism`, проверить `HasPassedCheck(NormControl)`.
2. **Блокировка загрузки после зачёта** — после "Зачёт" по нормоконтролю запретить загрузку новых версий работы. Валидация в `UploadAttachmentCommandHandler`.
3. **GitHub-ссылка для проверки ПО** — добавить поле `RepositoryUrl` к StudentWork или к QualityCheck(SoftwareCheck) для хранения ссылки на репозиторий.
4. **Цикл доработки** — при "Незачёт" по антиплагиату студент повторно проходит нормоконтроль, затем антиплагиат. Нужна валидация, что при повторном SubmitForCheck(AntiPlagiarism) нормоконтроль тоже пройден заново (новая попытка).
5. **Рецензия — полный цикл**:
   - Запрос `GetAssignedReviewerQuery` — студент видит назначенного рецензента.
   - Запрос `GetDocumentPackageQuery` — студент скачивает пакет документов для рецензента (работа + отзыв НР + справки проверок).
   - Мониторинг рецензий кафедрой — `GetReviewStatusByDepartmentQuery`.

---

## Этап 10: Подготовка к защите

**Реализовано:** CreateCommission (GAK), CreateDefenseSchedule, AssignWorkToSlot.

**Не хватает:**
1. **Дашборд готовности** — `GetDefenseReadinessQuery`, показывающий для каждого студента статус по всем этапам: предзащиты ✅/❌, нормоконтроль ✅/❌, ПО ✅/❌, антиплагиат ✅/❌, рецензия ✅/❌.
2. **Отправка напоминаний** — команда `SendReadinessRemindersCommand` — массовая отправка уведомлений студентам с незавершёнными этапами, их НР, и экспертам с pending-проверками.
3. **Формирование списка допущенных** — `GetAdmittedStudentsQuery` — список студентов, прошедших все этапы, для документа PDF.
4. **Автоматическое создание слотов защиты** — `GenerateDefenseSlots` — создание 45-минутных слотов (аналогично предзащите, но для GAK).

---

## Этап 11: Защита

**Реализовано:** SubmitGrade, GenerateProtocol, FinalizeDefense, MarkAsDefended.

**Не хватает:**
1. **Загрузка материалов защиты** — валидация типов вложений для защиты (презентация, демо-материалы). Можно добавить новые `AttachmentType` (Demo, Handout).
2. **Предварительное согласование оценок** — аналогично предзащите, `PreliminaryDefenseReconciliationCommand`.
3. **Итоговая оценка** — при финализации защиты рассчитать итоговую оценку (отлично/хорошо/удовл/неудовл) на основе средних баллов комиссии и записать в `StudentWork.FinalGrade`.
4. **Уведомление студентов о результатах** — при FinalizeDefense отправить уведомление с итоговой оценкой.
5. **Архивация** — переход StudentWork в финальное состояние "Defended" или "Failed", блокировка дальнейших изменений.

---

## Сквозные задачи (Cross-cutting)

1. **Уведомления через доменные события** — создать/дополнить обработчики доменных событий (`INotificationHandler`) для ключевых событий с отправкой уведомлений через `INotificationService`.
2. **Валидация периодов** — интегрировать `IPeriodValidationService` во все команды, где операции привязаны к срокам (создание направлений, тем, заявок, проверок и т.д.).
3. **Расширение WorkflowStage enum** — добавить стадии для предзащит и проверок в enum `WorkflowStage`, если их нет.

---

## Рекомендуемый порядок реализации

1. Этап 3 → Этап 4 → Этап 5 → Этап 6 (цепочка тем/направлений)
2. Этап 7 → Этап 8 (предзащиты)
3. Этап 9 (проверки)
4. Этап 10 → Этап 11 (защита)
5. Сквозные задачи (уведомления, периоды) — параллельно с каждым этапом
