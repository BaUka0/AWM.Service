## 1. Организационная структура и Мультитенантность

Система спроектирована для обслуживания множества университетов в рамках одной инсталляции базы данных. Изоляция данных обеспечивается иерархической структурой с корневым элементом «Университет».

### Схема `Org` (Organization)

Описывает физическую структуру учебных заведений.

* **`Universities` (Университеты)**
  * Корневая сущность. Является **Тенантом**.
  * Каждая запись представляет отдельный вуз. Все дочерние сущности (пользователи, работы, приказы) жестко привязаны к `UniversityId`.

* **`Institutes` (Институты/Факультеты)**
  * Структурные подразделения внутри университета.

* **`Departments` (Кафедры)**
  * Основная функциональная единица, где происходит управление дипломными работами.
  * Большинство ролей и процессов замыкаются на уровень Кафедры.

### Схема `Common` (Общие справочники)

* **`AcademicYears` (Учебные года)**
  * Определяет временные рамки (например, «2025-2026»).
  * **Механизм архивации:** Поле `IsArchived` переводит все данные, связанные с этим годом, в режим «Только для чтения» (Read-Only).

* **`Periods` (Периоды) NEW v4**
  * Управление временными рамками для всех этапов workflow.
  * Примеры: 'DirectionSubmission', 'TopicCreation', 'TopicSelection', 'PreDefense1', 'PreDefense2', 'PreDefense3', 'FinalDefense'.
  * Кафедра устанавливает даты начала и окончания для каждого этапа.
  * Система блокирует подачу заявок/тем вне установленных периодов.

* **`NotificationTemplates` (Шаблоны уведомлений) NEW v4**
  * Предопределенные шаблоны для всех типов уведомлений.
  * Поддержка мультиязычности (kaz, rus, eng).
  * Примеры событий: 'DirectionApproved', 'ApplicationReceived', 'DefenseScheduled'.

* **`Notifications` (Уведомления) NEW v4**
  * История всех уведомлений пользователей.
  * Привязка к сущностям (Direction, Topic, Work, Application).
  * Отслеживание статуса прочтения.

## 2. Модель безопасности и Ролевая политика (RBAC)

Система использует модель **Context-Aware RBAC** (Ролевая модель, зависящая от контекста). Это означает, что права пользователя определяются не только его ролью, но и тем, *где* и *когда* эта роль применяется.

### Схема `Auth` (Authentication & Authorization)

* **`Users` (Пользователи)**
  * Хранит учетные данные (Логин, Хеш пароля, Email) и идентификаторы SSO (`ExternalId`).
  * Один пользователь (физическое лицо) имеет одну учетную запись, даже если он учится в магистратуре и работает лаборантом одновременно.

* **`Roles` (Роли)**
  * Справочник функциональных ролей: `Student`, `Supervisor`, `HeadOfDepartment`, `Secretary`, `Expert`, `Admin`.
  * Поле `ScopeLevel` определяет уровень применимости роли (Глобально, Университет, Кафедра, Комиссия).

* **`UserRoleAssignments` (Матрица назначений)**
  * Ключевая таблица безопасности. Связывает `User` + `Role` + `Context`.
  * **Принцип работы:** Пользователь получает права роли только при совпадении контекста.
  * *Пример:* Пользователь А имеет роль «Научный руководитель» с контекстом `DepartmentId = 10`. Он может управлять работами на кафедре №10, но для кафедры №11 он — обычный пользователь без прав.
  * Поле `AcademicYearId` позволяет назначать роли временно (например, «Член комиссии только на 2025 год»).

## 3. Образовательный профиль

Разделение учетной записи (Auth) и профиля (Education) позволяет системе корректно обрабатывать данные о статусе человека в вузе.

### Схема `Edu` (Education)

* **`Students` (Студенты)**
  * Профиль обучающегося. Содержит данные о текущем курсе, группе.
  * Связан с `AcademicPrograms` (Образовательные программы), что позволяет системе автоматически валидировать возможность выхода на защиту.

* **`Staff` (Сотрудники)**
  * Профиль преподавателя. Содержит данные об ученой степени, должности.
  * Поле `MaxStudentsLoad` задает техническое ограничение на количество курируемых дипломников.

## 4. Ядро процессов: Workflow Engine (Машина состояний)

Вместо жестко запрограммированной логики используется конфигурируемая машина состояний. Это позволяет гибко настраивать этапы прохождения работы через базу данных.

### Схема `Wf` (Workflow)

* **`WorkTypes` (Типы работ)**
  * Классификатор: «Курсовая работа», «Дипломная работа (Бакалавр)», «Магистерская диссертация», «Докторская диссертация».
  * Разные типы работ могут иметь разные цепочки статусов.

* **`States` (Состояния)**
  * Атомарные этапы жизненного цикла работы.
  * Примеры для работ: `Draft`, `OnReview`, `AntiPlagiarism`, `Approved`, `Defended`.
  * **NEW v4:** Состояния для направлений: `DirectionDraft`, `DirectionSubmitted`, `DirectionApproved`, `DirectionRejected`, `DirectionRequiresRevision`.
  * Поле `IsFinal` указывает на завершение процесса.

* **`Transitions` (Переходы)**
  * Ориентированный граф, описывающий разрешенные действия.
  * Запись вида: `FromState` → `ToState` + `AllowedRole`.
  * **Логика:** Перевести работу из статуса «На проверке» в статус «Утверждено» может только пользователь с ролью, указанной в `AllowedRole`.

## 5. Управление дипломными работами (Thesis Management)

Центральный модуль системы, где хранятся данные о работах студентов.

### Схема `Thesis`

#### **`Directions` (Направления ДП/ДР) NEW v4**
  * **Назначение:** НР создает направления перед темами (Этап 3 воркфлоу).
  * Содержат описания на трёх языках (`TitleRu`, `TitleEn`, `TitleKz`).
  * Имеют workflow: Draft → Submitted → Approved/Rejected/RequiresRevision.
  * **Temporal Table:** Все изменения автоматически фиксируются в `DirectionsHistory`.
  * **Связи:**
    - Привязаны к кафедре (`DepartmentId`)
    - Привязаны к НР (`SupervisorId`)
    - Привязаны к году (`AcademicYearId`)
  * **Важно:** Темы могут создаваться только в рамках утвержденных направлений.

#### **`Topics` (Банк тем) MODIFIED v4**
  * Темы, предлагаемые преподавателями.
  * Содержат описания на трёх языках (`TitleRu`, `TitleEn`, `TitleKz`).
  * **NEW v4:** Привязка к направлению (`DirectionId`).
  * **NEW v4:** Поддержка командных работ (`MaxParticipants` от 1 до 5).
  * Имеют флаг `IsApproved` (прошли ли валидацию на кафедре).
  * **Temporal Table:** История изменений сохраняется автоматически.

#### **`TopicApplications` (Заявки студентов) NEW v4**
  * **Назначение:** Механизм подачи заявок на темы (Этап 5 воркфлоу).
  * Студент подает заявку, НР принимает/отклоняет.
  * **Поля:**
    - `MotivationLetter` - мотивационное письмо студента
    - `Status` - Submitted/Accepted/Rejected
    - `ReviewComment` - комментарий НР при отклонении
  * **Важно:** Студент может подавать заявки на несколько тем, но работа создается только после принятия одной заявки.

#### **`StudentWorks` (Работы студентов) MODIFIED v4**
  * Главная транзакционная таблица.
  * Связывает: Тему + Учебный год + Кафедру.
  * Хранит `CurrentStateId` — текущее положение работы в машине состояний.
  * **NEW v4:** Денормализованное поле `DepartmentId` для быстрых запросов.
  * **BREAKING CHANGE v4:** Убрана прямая связь `StudentId` (теперь через `WorkParticipants`).
  * **Temporal Table:** Все изменения фиксируются автоматически.

#### **`WorkParticipants` (Участники работы) NEW v4**
  * **Назначение:** Поддержка командных работ (M:M связь).
  * Одна работа может иметь до 5 участников.
  * **Поля:**
    - `Role` - Leader (руководитель команды) или Member
    - `JoinedAt` - дата присоединения к проекту
  * **Constraint:** Один студент может участвовать в работе только один раз (`UQ_Work_Student`).

#### **`WorkflowHistory` (Журнал операций)**
  * Логирует каждое изменение статуса работы: кто изменил, когда, какой комментарий оставил.
  * Обеспечивает прозрачность бизнес-процесса.

#### **`Attachments` (Файловое хранилище) MODIFIED v4**
  * Хранит метаданные файлов: имя, тип, ссылку на хранилище (File path).
  * **Binary Data:** Сами файлы в БД не хранятся (используется внешнее хранилище S3/FileServer).
  * **Безопасность:** Поле `FileHash` (SHA-256) защищает от подмены файлов после загрузки.
  * Привязка `StateId` фиксирует, на каком этапе был загружен файл.
  * **Типы файлов:** 'Draft', 'Final', 'Presentation', 'Software'.

#### **`Experts` (Эксперты) NEW v4**
  * **Назначение:** Назначение преподавателей кафедры экспертами по типам проверок.
  * **Типы экспертизы:**
    - `NormControl` - Нормоконтроль
    - `SoftwareCheck` - Проверка ПО
    - `AntiPlagiarism` - Антиплагиат
  * Один преподаватель может быть экспертом по нескольким типам.

#### **`QualityChecks` (Проверки) MODIFIED v4**
  * Результаты проверок (Нормоконтроль, ПО, Антиплагиат).
  * **NEW v4:** Поле `AttemptNumber` для отслеживания цикла доработки.
  * **NEW v4:** `DocumentPath` для хранения подписанного документа антиплагиата.
  * **Workflow цикла:**
    1. Студент загружает материалы
    2. Эксперт проверяет
    3. Если `IsPassed = 0`, студент дорабатывает, `AttemptNumber++`
    4. Цикл повторяется до `IsPassed = 1`

#### **`Reviewers` (Рецензенты) NEW v4**
  * **Назначение:** База внешних рецензентов (не сотрудников вуза).
  * Содержит: ФИО, должность, ученая степень, организация, контакты.
  * Кафедра назначает рецензентов для каждой работы.

#### **`Reviews` (Рецензии) NEW v4**
  * Рецензии внешних экспертов.
  * **Поля:**
    - `ReviewText` - текст рецензии
    - `FileStoragePath` - скан подписанной рецензии с печатью
  * Студент отправляет материалы рецензенту, получает рецензию, загружает в систему.

#### **`SupervisorReviews` (Отзывы НР) NEW v4**
  * Отзывы научных руководителей о работах.
  * Являются обязательным документом для защиты.
  * Могут быть как текстовыми, так и в виде подписанного файла.

## 6. Защита и Оценивание

Модуль обеспечивает планирование защит и гибкую систему выставления оценок.

### Схема `Defense`

#### **`Commissions` (Комиссии) MODIFIED v4**
  * Группы экспертов (ГАК, комиссии предзащиты).
  * **NEW v4:** Поле `PreDefenseNumber` для разделения комиссий предзащит (1, 2, 3).
  * Имеют привязку к Кафедре и Году.
  * **Типы:** 'PreDefense', 'GAK'.

#### **`CommissionMembers` (Члены комиссии)**
  * Состав комиссии. Включает роли внутри комиссии (Председатель, Секретарь, Рядовой член).

#### **`Schedules` (Расписание)**
  * Временные слоты (Time slots) для защиты.
  * Связывает Комиссию и Студенческую работу с конкретной датой и аудиторией.

#### **`PreDefenseAttempts` (Попытки предзащит) NEW v4**
  * **Назначение:** Отслеживание всех попыток предзащит (1, 2, 3).
  * **Поля:**
    - `PreDefenseNumber` - номер предзащиты (1, 2, 3)
    - `AttendanceStatus` - явка (Attended, Absent, Excused)
    - `AverageScore` - средний балл от комиссии
    - `IsPassed` - прошел ли предзащиту
  * **Логика:** Студент, не прошедший предзащиту 1, направляется на предзащиту 2. При провале предзащиты 2 → предзащита 3 (пересдача).

#### **`EvaluationCriteria` (Критерии оценки - EAV)**
  * Справочник критериев оценивания (например, «Актуальность», «Качество доклада»).
  * Позволяет настраивать разные критерии и веса баллов для разных типов работ без изменения структуры таблиц.

#### **`Grades` (Оценки) TEMPORAL**
  * Таблица фактов оценивания.
  * Запись: `Член комиссии` поставил `Балл` за `Критерий` в рамках конкретной `Защиты`.
  * Итоговый балл рассчитывается агрегацией этих записей.
  * **Temporal Table:** Защита от изменения оценок задним числом.

#### **`Protocols` (Протоколы) MODIFIED v4**
  * Финализированный документ защиты.
  * **NEW v4:** Поля `ProtocolNumber` и `ProtocolDate` для официального учета.
  * Содержит итоговую оценку, решение комиссии и флаги подписания.

## 7. Аудит и Целостность данных

### 7.1. Системное версионирование (Temporal Tables)

Для критически важных таблиц (`StudentWorks`, `Topics`, `Directions`, `Grades`) используется механизм **System-Versioned Temporal Tables**.

* Любая операция `UPDATE` или `DELETE` приводит к автоматическому сохранению предыдущей версии строки в историческую таблицу.
* Это позволяет восстановить состояние системы на любой момент времени в прошлом (Time Travel).
* **История таблиц:**
  - `Thesis.DirectionsHistory`
  - `Thesis.TopicsHistory`
  - `Thesis.StudentWorksHistory`
  - `Defense.GradesHistory`

### 7.2. Мягкое удаление (Soft Delete)

* Для бизнес-сущностей (Заявки, Работы) физическое удаление запрещено логикой. Вместо этого работа переводится в статус `Cancelled` через State Machine.
* Для справочных данных используется флаг `IsActive` или `IsArchived`.
* Физическое удаление данных используется только для ошибочно созданных технических записей и контролируется механизмом Temporal Tables.

## 9. Миграция с v3 на v4

### Критичные изменения:

1. **Удален constraint `UQ_Student_Year_Work`**
   - В v3: студент мог иметь только одну работу в год
   - В v4: студент может участвовать в нескольких работах через `WorkParticipants`

2. **Добавлено поле `DepartmentId` в `StudentWorks`**
   - Денормализация для производительности
   - При миграции нужно заполнить из `Topics.DepartmentId`

3. **Изменено поле `ExpertId` → `AssignedExpertId` в `QualityChecks`**
   - Переименование для ясности

### Скрипт миграции:

```sql
-- 1. Создать WorkParticipants из существующих StudentWorks
INSERT INTO [Thesis].[WorkParticipants] ([WorkId], [StudentId], [Role])
SELECT [Id], [StudentId], 'Leader'
FROM [Thesis].[StudentWorks]
WHERE [StudentId] IS NOT NULL;

-- 2. Заполнить DepartmentId в StudentWorks
UPDATE sw
SET sw.[DepartmentId] = t.[DepartmentId]
FROM [Thesis].[StudentWorks] sw
INNER JOIN [Thesis].[Topics] t ON sw.[TopicId] = t.[Id];

-- 3. Создать направление по умолчанию для существующих тем
INSERT INTO [Thesis].[Directions] 
    ([DepartmentId], [SupervisorId], [AcademicYearId], [WorkTypeId], 
     [TitleRu], [CurrentStateId])
SELECT DISTINCT 
    t.[DepartmentId], 
    t.[SupervisorId], 
    t.[AcademicYearId], 
    t.[WorkTypeId],
    'Прочее (миграция)',
    (SELECT [Id] FROM [Wf].[States] WHERE [SystemName] = 'DirectionApproved')
FROM [Thesis].[Topics] t
WHERE t.[DirectionId] IS NULL;

-- 4. Связать темы с направлениями
UPDATE t
SET t.[DirectionId] = d.[Id]
FROM [Thesis].[Topics] t
INNER JOIN [Thesis].[Directions] d ON 
    t.[DepartmentId] = d.[DepartmentId] AND
    t.[AcademicYearId] = d.[AcademicYearId];
```

## 10. Пример использования

### Сценарий: Студент подает заявку на командную тему

```sql
-- 1. НР создает направление
INSERT INTO [Thesis].[Directions] 
    ([DepartmentId], [SupervisorId], [AcademicYearId], [WorkTypeId],
     [TitleRu], [TitleKz], [Description], [CurrentStateId])
```

## 11. Часто задаваемые вопросы (FAQ)

**Q: Почему убрали constraint `UQ_Student_Year_Work`?**  
A: Он блокировал командные работы. Теперь связь M:M через `WorkParticipants`.

**Q: Обязательно ли создавать направления?**  
A: Да, согласно воркфлоу НР сначала создает направления, кафедра утверждает их, затем НР создает темы в рамках направлений.

**Q: Как отследить, прошел ли студент все предзащиты?**  
A: Запрос к `PreDefenseAttempts` с группировкой по `PreDefenseNumber` и проверкой `IsPassed = 1`.

**Q: Можно ли иметь несколько попыток проверки нормоконтроля?**  
A: Да, поле `AttemptNumber` в `QualityChecks` отслеживает все попытки.

**Q: Как система блокирует подачу заявок вне периодов?**  
A: Бизнес-логика приложения проверяет `Common.Periods` перед созданием заявки. В БД это constraint не реализован (гибкость).
